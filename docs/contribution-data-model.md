# Contribution Data Model

When a user submits a scan, four things are written in sequence.

---

## 1. Source (`/rest/v1/sources`)

Represents the **physical inscription** — the stone stele, temple sign, manuscript, etc. One source can accumulate many scans from different contributors over time.

| Field | Type | Notes |
|-------|------|-------|
| `id` | UUID | Generated by Postgres |
| `source_type` | enum | `stone_stele`, `wooden_inscription`, `paper_manuscript`, `printed_book`, `temple_sign`, `rubbing`, `other` |
| `title` | text? | Optional user-provided title |
| `location_name` | text | Required — e.g. "Temple of Literature, Hà Nội" |
| `province` | text? | Optional region |
| `country` | text | Defaults to "Vietnam" |
| `condition` | enum? | `excellent`, `good`, `weathered`, `damaged`, `severely_damaged` |
| `estimated_period` | text? | Free text — e.g. "15th century" |
| `coordinates` | geometry? | PostGIS point — not yet collected by iOS (always null) |
| `contributor_device_id` | text | `UIDevice.identifierForVendor` UUID |
| `is_public` | bool | Set server-side, defaults true |
| `verified` | bool | Set server-side by moderators |

---

## 2. Scan (`/rest/v1/scans`)

Represents **this decode session** — one photo, one run of the ML pipeline.

| Field | Type | Notes |
|-------|------|-------|
| `id` | UUID | Generated by Postgres |
| `source_id` | UUID | FK → sources |
| `contributor_device_id` | text | Same device ID as source |
| `image_path` | text? | **Always null currently** — upload happens after row creation, URL not PATCHed back |
| `thumbnail_path` | text? | Always null currently |
| `character_count` | int | Count of all detected characters (including unclear) |
| `transliteration` | text? | Space-joined quoc ngu for every character that has one |
| `decode_results` | jsonb? | Inline array of every character result (see below) |
| `has_corrections` | bool | Always `false` on initial submission |

### `decode_results` JSONB schema (per element)

```json
{
  "index": 0,
  "unicode": "佛",
  "quoc_ngu": "phật",
  "confidence": "high",
  "script_type": "han"
}
```

Characters with no recognized unicode are excluded from this array (compactMap filters `r.character == nil`).

---

## 3. Image → Supabase Storage

Uploaded to the `scan-images` bucket at:
```
{sourceId}/{scanId}/original.jpg
```

- Format: JPEG at 90% quality
- Upload is **best-effort** — wrapped in `try?`, failure is non-fatal
- The resulting public CDN URL is returned by `uploadScanImage` but **not persisted** (see known gap in `scan-image-upload.md`)

---

## 4. Characters (`/rest/v1/characters`)

One row per detected character. This is the **normalized, corpus-queryable** form of the same data in `decode_results`. Bulk-inserted in a single POST.

| Field | Type | Notes |
|-------|------|-------|
| `scan_id` | UUID | FK → scans |
| `source_id` | UUID | FK → sources (denormalized for query convenience) |
| `position_index` | int | Zero-based order in the scan |
| `unicode_char` | text | The Han/Nôm character |
| `quoc_ngu` | text\|null | Vietnamese romanization |
| `meaning` | text\|null | English gloss |
| `confidence` | enum\|null | `high`, `medium`, `low` |
| `script_type` | enum\|null | `han`, `nom` |
| `is_corrected` | bool | Always `false` on initial submission |

### Why both JSONB and normalized rows?

| | `scans.decode_results` | `characters` table |
|---|---|---|
| Use case | Read back a single scan's full results fast | Query across the whole corpus |
| Example | "Show me this scan's characters" | "Find all scans containing 佛" |
| Keys | Always identical (explicit `encode(to:)`) | Individually indexed |

---

## Duplicate handling

- **Sources**: no deduplication — each submission creates a new row even if the same location is submitted twice.
- **Characters**: PostgREST `resolution=merge-duplicates` — inserts are upserted on the unique constraint (if one exists). Currently the response body is empty (201), handled by the `data.isEmpty` guard in `SupabaseClient.post`.
